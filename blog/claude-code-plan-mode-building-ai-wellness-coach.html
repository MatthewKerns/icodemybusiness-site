Building an AI Wellness Coach: From Burnout to a Personal Operating System

Matthew Kerns March 12, 2026 15 min read

What you're reading: This is a real Claude Code plan mode output. I described a problem, Claude mapped out the entire system architecture, and then I sent it off to build. This is the exact plan that became a working system.

The Problem

I'd been experiencing low energy and motivation from neglecting the basics — food, water, sunlight, walking, passion projects. Too much time passively consuming content. Not enough time doing the things that actually give you energy to do great work.

So I asked Claude Code to plan an AI-powered personal wellness management system. Here's what I wanted:

  1. A daily wellness briefing email each morning (separate from my existing business briefing)
  2. A link to a coaching chatbot that helps overcome objections to self-care
  3. Wellness tracking via forms (multiple choice + free form) and conversational check-ins
  4. A basic financial pulse — monthly revenue target, break-even tracking
  5. The coaching bot connects self-care to the bigger vision: take care of yourself → have energy → do amazing work → help people → financial reward

I opened Claude Code, entered plan mode, described what I wanted, and this is what came back.

Phase 1: Types & Data Model

wellness/types/wellness.types.ts — Core Data Types

// Wellness domain categories
type WellnessCategory =
  | 'nutrition'
  | 'hydration'
  | 'movement'
  | 'energy'
  | 'passion_project'
  | 'financial_pulse';

// Wellness check-in entry (one per day, can be updated)
interface WellnessCheckIn {
  date: string;              // ISO date
  userId: string;
  entries: WellnessEntry[];  // Individual category entries
  overallMood: MoodLevel;    // 1-5 scale
  freeFormNotes?: string;    // Anything else on their mind
  submittedAt: number;       // timestamp
  source: 'form' | 'chat' | 'email_reply';
}

type MoodLevel = 1 | 2 | 3 | 4 | 5;

// Individual wellness entry
interface WellnessEntry {
  category: WellnessCategory;
  data: NutritionData | HydrationData | MovementData
      | EnergyData | PassionProjectData | FinancialPulseData;
}

wellness/types/wellness-checkin.types.ts — Form Structure

Every check-in category has multiple choice fields (quick to fill) plus an optional free-form notes field (for context the bot can use later):

// Nutrition check-in (multiple choice + free form)
interface NutritionData {
  mealsEaten: 0 | 1 | 2 | 3;
  calorieEstimate: 'way_too_low' | 'low' | 'about_right' | 'high';
  groceriesStocked: 'empty' | 'running_low' | 'decent' | 'well_stocked';
  cookedToday: boolean;
  needsGroceryRun: boolean;
  notes?: string;
}

interface HydrationData {
  glasses: 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8;
  feeling: 'dehydrated' | 'ok' | 'good' | 'great';
  notes?: string;
}

interface MovementData {
  wentOutside: boolean;
  walkMinutes: number;
  sunlightMinutes: number;
  bedroomTimeHours: number;  // Hours spent lying in bed (not sleeping)
  notes?: string;
}

interface EnergyData {
  sleepHours: number;
  sleepQuality: 'terrible' | 'poor' | 'ok' | 'good' | 'great';
  energyLevel: 1 | 2 | 3 | 4 | 5;
  screenTimeHours: number;   // Non-work screen time
  notes?: string;
}

interface PassionProjectData {
  workedOnProject: boolean;
  minutesSpent: number;
  projectName?: string;
  whatDidYouDo?: string;
  notes?: string;
}

interface FinancialPulseData {
  monthlyTarget: number;
  currentRevenue: number;
  momentum: 'declining' | 'flat' | 'growing';
  notes?: string;
}

wellness/types/wellness-kpi.types.ts — Wellness KPI Set

12 KPIs across 6 categories (2 per category), all reusing the existing PersonalKPI interface:

import { PersonalKPI } from '../../types/kpi.types';

interface WellnessKPISet {
  // Nutrition (2)
  dailyMealCount: PersonalKPI;       // Target: 3 meals/day
  groceryReadiness: PersonalKPI;     // Target: decent or well_stocked

  // Hydration (2)
  dailyWaterIntake: PersonalKPI;     // Target: 8 glasses
  hydrationConsistency: PersonalKPI; // Target: ≥6 glasses for 5+ of last 7 days

  // Movement (2)
  dailyWalkMinutes: PersonalKPI;     // Target: ≥30 min
  outdoorDaysStreak: PersonalKPI;    // Target: went outside every day

  // Energy (2)
  sleepQualityScore: PersonalKPI;    // Target: good or great
  screenTimeControl: PersonalKPI;    // Target: ≤2 hours non-work

  // Passion Projects (2)
  weeklyProjectHours: PersonalKPI;   // Target: ≥5 hours/week
  projectConsistency: PersonalKPI;   // Target: worked on project ≥4 of last 7 days

  // Financial Pulse (2)
  monthlyRevenueProgress: PersonalKPI;  // Target: on pace for monthly target
  revenueDirection: PersonalKPI;        // Target: growing momentum
}

wellness/types/wellness-briefing.types.ts — Briefing Structure

interface WellnessBriefing {
  date: string;
  greeting: string;                 // Personalized, empathetic greeting
  wellnessScore: WellnessScore;     // Overall wellness snapshot
  kpis: WellnessKPISet;
  insights: WellnessInsight[];      // Actionable wellness insights
  coachingPrompt: string;           // Today's coaching nudge
  chatLink: string;                 // Link to coaching chatbot
  calendarInsight?: string;         // "You have free time at 2pm — perfect for a walk"
  visionReminder: string;           // Connection to bigger purpose
}

interface WellnessScore {
  overall: 'thriving' | 'doing_ok' | 'needs_attention' | 'struggling';
  categoryScores: Record<WellnessCategory, 'green' | 'yellow' | 'red'>;
  streakDays: number;               // Consecutive days of check-ins
}

interface WellnessInsight {
  category: WellnessCategory;
  priority: 'low' | 'medium' | 'high' | 'critical';
  message: string;                  // "You haven't been outside in 2 days"
  action: string;                   // "Just walk for 10 minutes right now"
  motivationalHook?: string;        // "You always feel better after even a short walk"
}

Phase 2: Data Storage & KPI Calculators

WellnessDataStore

  • Store daily check-in data (initially in-memory / JSON file, later Convex)
  • Retrieve check-in history for KPI calculations (last 7 days, last 30 days)
  • Update partial check-ins (user can log meals throughout the day)

KPI Calculators

One calculator per category, following the exact pattern of existing calculators like RockExecutionCalculator:

  • Each takes check-in history as input
  • Returns PersonalKPI objects with currentValue, targetValue, trend, status, insight
  • Thresholds defined per KPI (e.g., ≥3 meals = on_track, 2 = at_risk, ≤1 = off_track)

Phase 3: Insight Generation

Following the existing InsightGenerator pattern. Key insight templates:

  • Nutrition: "You've eaten {meals} meals today. Your fridge is {status}. {action}"
  • Movement: "You haven't been outside in {days} days. {nudge}" / "Great streak! {days} days of walking."
  • Energy: "You spent {hours}h on YouTube yesterday. That's {trend} from your average."
  • Passion: "{days} days since you last worked on {project}. Even 30 minutes counts."
  • Financial: "You're at ${current} of your ${target} target. {momentum_message}"

Each insight generator produces insights with a motivationalHook — a short statement connecting the action to the user's bigger vision.

Phase 4: The Coaching Agent

This is the heart of the system. The WellnessCoachingAgent system prompt encodes:

Core belief: Taking care of yourself is not optional — it's the foundation for everything you want to achieve.

User's vision: Help many people, build something incredible, achieve financial independence.

Objection handling framework:

  • "I don't feel like it" → "You never feel like it before. You always feel better after. Just start with 5 minutes."
  • "I'll do it tomorrow" → "You said that yesterday. Today IS the day. What's one small thing right now?"
  • "It doesn't matter" → "Your energy directly impacts your client work. Taking care of yourself IS your job."

Conversation style: Direct, compassionate, no-BS. Like a coach who genuinely cares.

Data-informed: References actual check-in data ("You've only had 1 meal today. That's why you feel foggy.")

Action-oriented: Always ends with a concrete next step, not just motivation.

Vision connection: Regularly connects back to: self-care → energy → great work → helping people → financial reward.

The agent will:

  1. Load the user's recent wellness data (last 7 days)
  2. Load today's briefing insights
  3. Use the LLM (via @agent-os/core LLM managers) with the coaching system prompt
  4. Handle conversation via both email replies and web chat
  5. Store conversation context for continuity

Phase 5: Briefing Service & Email Template

WellnessBriefingService

  • Mirrors DailyBriefingService orchestration pattern
  • Fetches wellness check-in data
  • Runs KPI extractors and insight generators
  • Checks calendar for available time slots (via existing CalendarBriefingHelper)
  • Generates personalized greeting and coaching prompt
  • Produces WellnessBriefing object

Email Template

  • HTML + plain text versions (same pattern as daily-briefing-email.ts)
  • Visual wellness score (color-coded categories)
  • Top 3 insights with action items
  • Chat link prominently placed
  • "Reply to this email to talk with your coach" instruction
  • Vision reminder at the bottom
  • Warm, personal tone — not corporate

Phase 6: Reply Processing & Chat Integration

  • Mirrors DailyBriefingReplyProcessor
  • Parses email replies using existing ReplyParser from @agent-os/core
  • Routes to WellnessCoachingAgent for response
  • Can extract check-in data from natural language: "I ate 2 meals, drank 4 glasses of water, went for a 20 min walk"

Phase 7: Export & Integration

Update packages/personal-agent-os/src/index.ts to export the wellness module.

Implementation Order

Dependency-driven — each step unlocks the next:

  1. Types (no dependencies) — All type files
  2. WellnessDataStore (depends on types) — Storage layer
  3. KPI Calculators (depends on types + data store) — All 6 calculators
  4. WellnessKPIExtractor (depends on calculators) — Orchestrates all calculators
  5. Insight Generators (depends on KPIs) — All 5 insight modules + aggregator
  6. WellnessCoachingAgent (depends on data store + insights) — Coaching bot
  7. WellnessBriefingService (depends on all above) — Main orchestrator
  8. Email Template (depends on briefing types) — HTML/text generation
  9. WellnessBriefingReplyProcessor (depends on coaching agent) — Email reply handling
  10. Integration (depends on all above) — Exports, wiring to email/chat systems

Verification Plan

  1. Unit tests for each calculator (check thresholds, edge cases, trend calculations)
  2. Unit tests for insight generators (verify correct insights for different data states)
  3. Unit tests for coaching agent (verify system prompt, objection handling)
  4. Integration test: Generate a full wellness briefing from mock check-in data
  5. Integration test: Generate email HTML and verify it renders correctly
  6. Integration test: Process an email reply and verify it routes to coaching agent
  7. Manual verification: Run npm run build and npm run check — no type errors
  8. Manual verification: Run npm run test — ≥85% coverage on new code

Key Design Decisions

  1. Reuse PersonalKPI interface — Don't create a new KPI type. The existing interface (currentValue, targetValue, trend, status, insight) works perfectly for wellness metrics.
  2. Separate briefing — Own email, own schedule, own thread. Clean domain separation.
  3. In-memory data store initially — Start simple. The check-in data can move to Convex later. Don't over-engineer storage before we have real usage patterns.
  4. Coaching prompt is the secret sauce — Invest heavily in crafting the system prompt. It needs to genuinely understand the user's psychology, not just be a generic wellness bot.
  5. Calendar awareness — Use existing CalendarBriefingHelper and AvailabilityCheckTool to find time for cooking, grocery shopping, and walks. This makes the system actually useful, not just nagging.
